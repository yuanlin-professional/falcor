# 错误处理

**注意：这些指南代表了未来推荐的错误处理方式。现有代码使用了多种错误处理方法，包括完全不检查错误。**

Falcor 及其应用程序必须能够以一致的方式处理开发和执行过程中发生的错误。
错误大致分为三大类，处理方式各不相同：

1. *逻辑错误*是不应该发生的情况；它们是程序员失误的结果。处理它们的主要机制是*断言*。
2. *运行时错误*是非预期的失败，但可能由不利的运行时条件或无效输入引起。处理它们的主要机制是抛出*异常*。
3. *可能发生的错误*，且通常需要检查并从失败中恢复。这些通过*返回值*或*返回码*处理。

异常是主要的错误处理方法，因为它相比其他方法（如返回码）具有以下优势：

* 不会因为忘记检查返回码而导致失败被忽略。异常会强制调用代码识别并处理错误。
* 异常沿调用栈向上传递，直到应用程序处理它或程序终止。中间层会自动传播错误。
* 异常的栈展开会按照明确定义的规则销毁作用域内的所有对象。
* 检测错误的代码和处理错误的代码之间有清晰的分离。
* 异常可以以明确定义的方式跨越 Python API 边界。

注意，错误报告是与错误处理不同的主题，将在后文讨论。所有发生的失败情况都必须被适当检测和处理。

## 断言

* 使用断言检查 C++ 代码中的逻辑错误；这些错误不应该发生，是程序员的失误而非运行时条件导致的。
* 如果可能，使用 `static_assert()` 在编译时检查逻辑错误，例如模板参数的类型检查或结构体大小检查。
* 使用 `FALCOR_ASSERT()` 在运行时检查逻辑错误。
* 使用 `FALCOR_ASSERT_EQ`、`FALCOR_ASSERT_NE`、`FALCOR_ASSERT_GE`、`FALCOR_ASSERT_GT`、`FALCOR_ASSERT_LE`、`FALCOR_ASSERT_LT` 对两个值进行二元比较。这些断言宏会打印消息，不仅显示测试条件，还显示导致失败的值。
* 大量使用断言。即使是显然正确的代码也可能受到其他地方更改的影响。
* 断言默认仅在 Debug 构建中启用。养成定期在 Debug 模式下运行应用程序的习惯，确保没有断言被触发。

## 异常

* 使用异常检查可能发生的运行时错误。例如，错误的用户输入、缺失或损坏的文件、内存不足等。
* 使用异常检查 shader 错误。Shader 程序是动态加载的，代码中的错误被视为运行时错误。
* 使用异常检查公共 API 函数的参数。即使你的代码是正确的，你也无法控制用户可能传入什么参数。
* 抛出异常时附带描述性消息。内置的异常类接受格式化字符串以简化此过程。
* 按值抛出异常，按引用捕获异常。
* 不要在用户代码中捕获异常，除非在极少数情况下。Falcor 的默认错误处理程序会记录错误并终止应用程序，这通常是最合理的操作。

Falcor 提供了自己的异常类（继承自 `std::exception`）和一些实用宏以改善开发体验。主要的异常类是 `Exception`。为了简单起见，我们尽量避免使用过多不同类别的异常。主要异常应始终使用 `FALCOR_THROW(fmt, ...)` 宏抛出，该宏支持格式化字符串来格式化错误消息。这个宏的优势在于，如果附加了调试器，它能够在调试器中中断。这在开发过程中很方便，因为我们会在抛出异常之前自动中断到调试器，使得查找导致异常的代码/条件变得容易。另外也可以使用调试器的"异常时中断"功能，但使用起来可能不太方便，因为它会在**所有**异常上中断，包括本地和外部代码。`FALCOR_THROW` 宏还会在没有附加调试器的情况下记录异常抛出位置的堆栈跟踪，有助于事后调试。

除了 `FALCOR_THROW` 之外，还有一个 `FALCOR_CHECK(cond, fmt, ...)` 宏，可用于检查条件，包括检查有效参数以及运行时不变量。该宏的定义很简单：

```c++
#define FALCOR_CHECK(cond, fmt, ...)        \
    if (!(cond))                            \
        FALCOR_THROW(fmt, ##__VA_ARGS__)
```

## 断言与异常的选择

有时使用*断言*还是*异常*的界限可能比较模糊。一般来说，所有面向公共的 API 都应严格使用*异常*，对于通过 Python API 暴露的函数尤其如此。内部代码可以更多地依赖断言。以下是一个简单示例：

```c++
// 这是面向公共的函数。
// 如果数据格式无效，我们抛出异常。
void processTriplets(std::span<float> data)
{
    FALCOR_CHECK(
        data.size() % 3 == 0,
        "'data' needs to contain multiples of 3 values but got size={}.",
        data.size()
    );
    processTripletsInternal(data)
}

// 这是内部函数。
// 我们假设数据格式正确，但仍然检查前置条件，
// 以帮助在开发/重构过程中发现逻辑错误。
void processTripletsInternal(std::span<float> data)
{
    FALCOR_ASSERT(data.size() % 3 == 0);
}
```

## 返回值

* 对于因非程序员错误而可能失败的操作，如果预期的常见情况是实际检查并处理失败，则应使用空返回值、错误码等。
* 此类情况的示例包括用户对话框、配置文件中的错误输入等。
* 在性能关键的紧密循环中，当检测和处理失败都在本地完成时，也可以使用返回值或错误码。例如，遍历顶点缓冲区检查无效顶点时。

## 错误报告

Falcor 提供了一些辅助工具用于向用户报告错误。请注意，除了警告之外，应用程序代码通常应优先抛出异常而非报告错误。框架会捕获这些异常并相应地报告它们。在某些情况下，应用程序可能希望在错误条件下等待用户输入，例如脚本执行失败时，允许用户修复脚本并重试。但这些情况通常很少见。

### 警告

* 使用 `logWarning(msg)` 报告非关键但意外的情况。
* 例如，对于会负面影响性能但不会影响执行的情况，使用警告可能是合适的。
* 警告会打印到控制台、调试器输出窗口和日志文件（取决于启用的日志输出），但很容易被忽略。除非用户明确查看，否则不要依赖用户看到警告消息。

### 错误

* 使用 `reportError(msg)` 报告关键错误，用户可以选择继续执行。
  * 消息以 `Error` 级别记录。
  * 显示消息框（除非已禁用），用户可以选择中止（终止应用程序）、进入调试器（如果已附加）或继续。
* 使用 `reportErrorAndAllowRetry(msg)` 报告关键错误，用户可以选择重试操作。例如，shader 编译失败时使用此函数。
  * 消息以 `Error` 级别记录。
  * 显示消息框（除非已禁用），用户可以选择中止（终止应用程序）、进入调试器（如果已附加）或重试。
* 使用 `reportFatalError(msg)` 报告致命错误，应立即终止应用程序以避免任何未定义行为。
  * 消息以 `Fatal` 级别记录。
  * 显示消息框（除非已禁用），用户可以选择中止（终止应用程序）或进入调试器（如果已附加）。

如果使用 `setShowMessageBoxOnError(false)` 禁用了消息框，上述所有 `reportError` 函数会在记录错误后立即终止应用程序。

不要在抛出异常之前调用任何 `reportError` 函数，因为默认的异常处理程序已经会报告错误。

## 日志

Falcor 提供了日志基础设施。使用以下全局函数记录消息：`logDebug`、`logInfo`、`logWarning`、`logError` 和 `logFatal`。

### 级别

Falcor 使用以下指南来选择不同的日志级别：

| 级别 | 描述 |
| --- | --- |
| `Debug` | 可能需要用于诊断问题和故障排除的消息。 |
| `Info` | 纯信息性的消息。此级别不应用于指示任何意外情况。 |
| `Warning` | 表示发生了意外情况，但应用程序能够继续运行的消息。 |
| `Error` | 表示发生了错误，应用程序可能无法继续正确运行的消息。 |
| `Fatal` | 表示发生了致命错误，应用程序需要立即终止的消息。 |

### 详细程度

日志记录器的详细程度可以通过设置消息记录的级别上限来配置。默认设置为 `Logger::Level::Info`，即除 `Debug` 以外的所有级别都会被记录。可以使用 `Logger::setVerbosity` 更改详细程度。

### 输出流

日志系统可以向以下输出流记录日志：

- 控制台（stdout）
- Visual Studio 调试窗口（如果已附加调试器）
- 文件

默认情况下，三个输出流都处于启用状态。可以使用 `Logger::setOutputs` 更改此设置。

当记录到文件时，日志记录器会根据执行进程的名称和每次启动进程时递增的编号自动选择文件名。对于 `Mogwai.exe`，会生成名为 `Mogwai.exe.0.log`、`Mogwai.exe.1.log` 等的日志文件。

**注意**：Falcor 4.4 及更早版本使用日志记录器在错误条件下弹出对话框或允许用户重试操作。在当前版本中，日志记录器仅用于记录消息，不附加其他逻辑。

## Falcor 用户指南

### 资源分配

* 资源分配失败是运行时错误，Falcor 将在内部抛出异常。
* 用户代码可以假定资源分配成功，不需要检查 `nullptr`。

### Shader 程序创建

* Shader 编译错误在 Falcor 内部处理。
* 默认行为是询问用户如何继续，给用户修正 shader 并重试的机会。
* 如果编译失败且用户选择中止/取消，则为运行时错误，Falcor 内部会抛出异常。
* 用户代码可以假定 shader 程序创建成功，不需要检查 `nullptr`。

### Shader 绑定

* Shader 绑定代码应假定变量存在于 shader 程序中。
* 如果不存在，则为运行时错误，Falcor 内部会抛出异常。
* 用户代码不需要检查返回值，因为错误已经被处理。
* 如果不确定变量是否存在，应先使用 shader 反射 API 进行查询。

### Shader 反射

* 失败是 shader 反射的预期结果，因为它用于查询 shader 变量是否存在。
* 因此，shader 反射 API 使用返回值来表示成功或失败，例如，如果找不到类型，则返回 `nullptr` 而不是类型反射对象。
* 用户代码必须检查 shader 反射调用的返回值。
